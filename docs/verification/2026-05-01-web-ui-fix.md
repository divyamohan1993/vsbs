### Path to live CARLA

The existing live CARLA bridge ([`tools/carla/vsbs_carla/scripts/run_demo_live.py`](../../tools/carla/vsbs_carla/scripts/run_demo_live.py)) drives a real Tesla Model 3 ego in `Town10HD`. It already POSTs to `/v1/sensors/ingest`. Adding the L5 dashboard feed is now a one-line wire: anywhere it computes a frame, `await vsbs_api.autonomy_telemetry(booking_id, frame)` and the dashboard sees CARLA-truth instead of the deterministic synth.

`run_chaos_demo.py` is the wire-identical scenario driver: same booking-id, same SSE shape, same event categories, same severities. A consumer can't distinguish the two — by design, so we can iterate the dashboard without booking GPU time.

### Honest constraint

This dev box is a 19 GiB / 2 GiB-VRAM laptop GPU (NVIDIA 940MX). CARLA 0.9.16 even in `-RenderOffScreen -quality-level=Low` typically wants 4 – 6 GiB VRAM and ~12 GiB RAM. A previous launch attempt this session brought the API and Next.js dev server down via OOM. The chaos driver exists precisely so the autonomy dashboard demo runs end-to-end on this class of hardware. On a workstation with ≥ 6 GiB free VRAM, swap the chaos driver for `vsbs-carla-demo` and the dashboard reads the live CARLA world without any further changes.
