## Headline result

All five runtime-reproducible findings from the audit's PoC (`runtime-probes.ts`) are sealed. Forged-IAP admin reaches 401 instead of 200; spoofed `x-vsbs-owner` reaches 401; unauth payment-order create reaches 401; sim-signed autonomy grant requires session; public concierge thread read reaches 401; public scenario consent bootstrap reaches 401. The OTP `/verify` route now mints an HMAC-SHA-256 signed bearer (`OtpVerifyResponse.sessionToken` + `expiresAt`) consumed by every owner-scoped route via the new `requireSession` middleware. The admin gate verifies real ES256 IAP signatures with cached JWKs. Production fail-closed is enforced by an `env.ts` `superRefine` that rejects the dev signing-key default + sim modes when `NODE_ENV=production`. Cloud Armor is attached to every backend service in Terraform; `/admin` traffic now flows through an IAP-protected backend. Mobile uses the new `sessionToken` instead of the raw OTP subject.

## Production fail-closed (Layer 1 contract check)

`apps/api/src/env.ts` `superRefine` now rejects the following at startup when `NODE_ENV=production`:
- `AUTH_MODE`, `PAYMENT_MODE`, `AUTONOMY_MODE`, `MERCEDES_IPP_MODE`, `MAPS_MODE`, `ADMIN_AUTH_MODE` ≠ `live`
- `LLM_PROFILE` = `sim`
- `SESSION_SIGNING_KEY` left at the dev default
- `IDENTITY_PLATFORM_SIGNING_KEY` left at the dev default
- `GCP_IAP_AUDIENCE` unset

12 new env vars on the API container get written by Terraform `infra/terraform/modules/region/main.tf`. `SESSION_SIGNING_KEY` is generated by `random_password` (64 chars) into Secret Manager. `GCP_IAP_AUDIENCE` is written after the admin backend exists to break the otherwise-cyclic dependency.

- IAP: `google_iap_brand` + `google_iap_client` in `infra/terraform/iap.tf`; `iap { enabled, oauth2_client_id, oauth2_client_secret }` on the new admin backend; URL map routes `/admin`, `/admin/*`, `/v1/admin`, `/v1/admin/*` through it. Members listed via `var.iap_admin_members`.

## Coordinator-applied fixes during verification

Two issues surfaced when integrating the three parallel wave-2 workstreams (`route-authz-1`, `route-authz-2`, `infra-and-edge`); patched directly:

1. **Concierge default-headers forwarding.** `route-authz-1` left a `// TODO(agent C)` marker at `apps/api/src/routes/concierge.ts` near the `buildVsbsGraph(...)` call; `route-authz-2` declined to edit B's file. The verified bearer from `requireSession` is now forwarded into the agent graph's `defaultHeaders` per request so any tool callback against authenticated endpoints carries identity. The cached graph was replaced with a per-request graph and a cached registry, so the registry cost is paid once.
2. **Sensor consent gate ordering.** `route-authz-2` mounted `requireConsent("diagnostic-telemetry")` at app level on `/v1/sensors/ingest`, ahead of the sensors-router-internal `optionalSession`. The gate fired `OWNER_REQUIRED` before the bearer could populate `ownerSubject`. Lifted the consent check inside the route handler, conditional on bearer auth (vehicle-token-authenticated producer ingest inherits consent through its booking-bound HMAC).

Both fixes pass the api package typecheck and the affected test suites.
